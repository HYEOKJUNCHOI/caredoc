## 3. 작업 원칙
- 추측 금지. 확인한 내용만 반영한다.
- 최소 변경. 요청 범위를 넘어서는 수정을 하지 않는다.
- 지정하지 않은 파일은 건드리지 않는다.
- 커밋은 반드시 컴파일 에러 없는 상태에서만 한다.
- 판단이 어려우면 무조건 질문한다. 추측으로 진행하지 않는다.
- 답변이 모호하면 즉시 재질문한다. 넘어가지 않는다.
- 지시가 짧거나 디테일이 부족하면 구체화 질문을 먼저 한 후 작업을 시작한다.

## 4. Communication
- **Language:** 모든 설명, 주석, 확인 요청, 계획 보고는 반드시 **한국어**로 작성한다.
- **Method:** 초보자도 이해할 수 있게 쉽게 설명하되, 구조와 로직을 명확히 짚는다.
- **Proactive:** 리스크나 더 나은 대안이 있으면 능동적으로 제안한다.
- **Format:** **[결론/해결책] → [코드] → [상세 설명]** 순서로 답변한다.
- **설명 포맷:**
  1. 코드 블록 먼저
  2. 구분선 `---`
  3. 영어 단어 한국어 발음/뜻 bullet: `* 단어(발음 : 뜻)`
- **확인 멘트:** 파일 수정 전 동의 요청 포함 모든 커뮤니케이션은 한국어로 한다.
  - 예: "BasicInfoDoc.jsx를 수정하려고 합니다. 진행할까요?"
- **🍞 추론 흔적:** 지시가 모호해서 추론했다면 응답 끝에 반드시 남긴다.
  ```
  🍞 추론 흔적: [무엇이 모호했는지] → [어떻게 쓰면 더 명확했을지]
  ```
  - 해석이 2가지 이상이고 확신이 없으면 추론하지 않고 바로 질문한다.

## 5. 레이아웃 변경 규칙

### 5-1. 최소 변경 원칙
- 명시하지 않은 요소의 위치, 크기, 패딩, 정렬은 절대 변경하지 않는다.
- 기존 요소를 임의로 이동하거나 재배치하지 않는다.

### 5-2. 기호 체계

| 기호 | 이름 | 의미 |
|------|------|------|
| `[]` | 앵커 | 이미 있는 요소. 건드리지 않는다. 위치 참조용. |
| `<>` | 포인터 | 삽입 위치. "여기에 넣어라". |
| `{}` | 이동 | 다른 곳에 있는 요소를 여기로 옮긴다. |
| `{+}` | 생성 | 새로 만들어서 삽입한다. |
| `{*}` | 부유 배치 | position + z-index로 띄워서 겹치게 배치한다. |

**조합 규칙:**
- `<{A}>` → A를 여기로 이동
- `<{+A}>` → A를 새로 만들어서 삽입
- `<{A,B}>` → A와 B를 함께 이동
- `<{A,+B}>` → A는 이동, B는 새로 생성하여 함께 삽입

**픽셀 넛지:**
- `{요소} -Npx` → 왼쪽으로 N픽셀 이동
- `{요소} +Npx` → 오른쪽으로 N픽셀 이동

### 5-3. 대규모 재구성 시
1. 즉시 실행하지 않고 변경 범위와 영향받는 요소 목록 먼저 보고한다.
2. 사용자 승인 후에만 실행한다.

## 6. 협업 프로토콜

### 역할 분담
- **CC (Claude Code):** 메인 개발 (코드 구현 전반, 약 80%)
- **AG (Antigravity):** Firebase·Google 서비스 연동, 문서/이미지 분석, CC 쿼터 교대 (약 20%)

### 동기화 규칙
- 1~6번 항목에 새로운 내용이 채워질 때마다 GEMINI.md도 동일하게 업데이트한다.
- 7번(워크플로우)은 각자 다르므로 동기화 대상에서 제외한다.
- 두 파일은 7번을 제외하고 항상 동일한 상태를 유지한다.
- 핸드오프 전 GEMINI.md 최신 상태 확인 필수.

### 토스 규칙
- 작업 중 상대 에이전트 적합 영역 감지 시 이유와 함께 토스 여부를 혁준님께 묻는다.
  - 예: "Firebase 연동 작업입니다. AG로 진행할까요?"
- 승인 시 핸드오프 절차 진행 후 대기한다.

### 핸드오프 규칙
- 사용자가 **"핸드오프 해"** 라고 지시하면 즉시 수행한다.
- AI가 스스로 판단해서 핸드오프하지 않는다.
- 핸드오프 절차:
  1. 현재 작업을 컴파일 에러 없는 상태까지 마무리한다.
  2. HANDOFF.md 업데이트한다.
  3. git add → commit → push 수행한다.

### 인수 규칙 (AG에게서 인수받을 때)
- CLAUDE.md + HANDOFF.md를 읽고 시작한다.
- 첫 진입이면 전체 코드를 훑으며 CLAUDE.md와 대조해 업무를 파악한 후 작업을 시작한다.

### 브랜치 전략
```
main
  └── develop
        ├── feature/cc-기능명
        ├── feature/ag-기능명
        └── fix/cc-버그명
```

### 커밋 메시지
```
[CC] feat: 설명
[AG] fix: 설명
```

| 타입 | 의미 |
|------|------|
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `style` | 스타일 변경 |
| `refactor` | 리팩토링 |
| `docs` | 문서 수정 |
| `chore` | 설정, 빌드 등 |

## 7. 워크플로우 (1인 4역)

<constraints>
- 추측 금지
- 최소 변경
- 기존 구조 유지
- 테스트 우선
</constraints>

<workflow>
1. 탐색
2. 계획
3. 구현
4. 검수
5. 승인
</workflow>

<output>
- 먼저 계획
- 그다음 변경안
- 마지막에 검증 결과
</output>

<approval>
- 검수와 비판이 모두 통과해야 완료
</approval>

**[기획]**
- 관련 파일과 문맥을 읽고 문제를 재정의한다.
- 목표, 제약, 성공 기준, 변경 범위를 짧게 정리한다.
- 아직 코드를 작성하지 마라.

**[개발]**
- 기획에서 정한 범위만 구현한다.
- 과설계하지 말고 최소 변경으로 해결한다.
- 추측하지 말고 확인한 내용만 반영한다.

**[검수]**
- 누락, 버그, 엣지 케이스, 회귀 가능성을 찾는다.
- 문제가 있으면 구체적으로 수정 지점을 지적한다.

**[비판]**
- 최종안의 약점을 가장 엄격하게 검토한다.
- 요구사항, 유지보수성, 성능, 안정성까지 본다.
- 승인 조건을 모두 만족할 때만 OK를 내린다.

**작업 규칙:**
- 단순 수정(오타, 색상, 텍스트 변경 등)은 바로 구현한다.
- 로직 변경, 구조 변경, 신규 기능은 반드시 4단계를 거친다.
- 판단이 애매하면 혁준님께 묻는다.
- 비판이 OK를 내리기 전까지 최종 결과를 제출하지 마라.
- 매 라운드마다 변경 이유를 한 줄로 요약하라.
- 가능하면 테스트/검증 명령까지 포함해서 끝내라.
